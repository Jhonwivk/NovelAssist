# AI 辅助小说创作系统 最终优化方案 v3

> 在原始大纲基础上做"重新抽象 + 深挖核心 + 调整顺序"。功能不丢，但围绕命门重新组织。

---

## 一、项目定位

**一句话**：面向网文/长篇作者的 AI 创作工作台，核心价值是**百万字级长篇的设定一致性与长程记忆管理**。

**差异化三角**（避免和通用 LLM、同类工具同质化）：

- **vs ChatGPT/Claude**：结构化 Story Bible + 长程记忆 + 写作工作流，不是"对话窗口"
- **vs 蛙蛙写作 / 墨狐 / Sudowrite**：真正可用的一致性引擎（不是 LLM 兜底问一句）
- **vs 人工审稿**：实时、低成本、可逐章定位问题

**北极星指标**：周活作者人均 AI 协作"有效字数"（被用户保留入正文）。

**主动不做**：纯文学严肃作者、随便玩玩的爱好者市场。

---

## 二、目标用户（聚焦 3 类）

1. **网文新人作者**：开新书 + 前 30 章存稿，付费意愿强
2. **百万字大长篇连载作者**：设定崩坏、伏笔遗忘是真痛点
3. **剧本杀 / 游戏剧情策划**：多结局 + 强人设约束

| 痛点 | 解决方案 |
|------|---------|
| 不知道写什么 | 灵感生成、题材组合、卖点设计 |
| 大纲难设计 | 三幕式/起承转合/网文升级流模板 |
| 长篇前后矛盾 | **一致性引擎 + Story Bible + 长程记忆** |
| 人物崩坏 | 角色卡 + 状态机 + 行为校验 |
| 伏笔忘记回收 | 伏笔状态机 + 自动提醒 |
| 文风不稳定 | 文风模板 + 样本学习 + 守卫 Agent |
| 修改困难 | 局部润色/改写/扩写/视角转换 |
| 投稿前质量不稳 | 审稿评分 + 钩子分析 + 节奏分析 |

## 三、核心数据资产：Story Bible（小说事实库）

把原大纲的角色 / 世界观 / 时间线 / 伏笔 / 道具 / 组织 / 地点 / 文风**统一抽象**：

```
Story Bible
├── Entities      实体（角色 / 地点 / 组织 / 道具 / 能力体系）
├── Relations     关系（实体间，含时间区间）
├── Events        事件（时间线节点 + 因果链）
├── States        实体时序状态快照（按章节维度）
├── Foreshadows   伏笔（特殊事件，含状态机）
├── Rules         世界观硬规则（一致性引擎弹药）
└── Style         文风特征 + 禁用词 + 偏好表达
```

**核心原则**：

- 所有 AI 调用基于 Story Bible 做 RAG
- 所有用户写作反向更新 Story Bible（自动事实抽取）
- 这是产品的**核心资产、壁垒和数据飞轮源头**

## 五、一致性引擎（核心章节）⭐

> 这是产品命门，单独深挖。原大纲只用一节带过，本方案升级为五层架构。

### 5.1 设计哲学

1. **分层防御**：能用规则解决的不调 LLM，能小模型解决的不调大模型
2. **可解释**：每条问题必须能定位章节/段落、关联实体、给出置信度
3. **可干预**：用户能"忽略此规则" / "标记为有意为之"，避免反复打扰

### 5.2 五层架构

```
┌─────────────────────────────────────────────┐
│ L5  反馈学习  采纳/忽略/有意 → 规则置信度调整  │
├─────────────────────────────────────────────┤
│ L4  LLM 语义  性格突变 / 动机不合理 / 文风跳脱│
├─────────────────────────────────────────────┤
│ L3  图谱推理  关系冲突 / 因果链 / 时序 / 可达 │
├─────────────────────────────────────────────┤
│ L2  规则引擎  等级单调 / 年龄 / 唯一性 / 持有 │
├─────────────────────────────────────────────┤
│ L1  事实抽取  正文 → 结构化 facts            │
└─────────────────────────────────────────────┘
```

L1-L3 零 LLM 成本、毫秒级；L4 才调模型；L5 形成数据飞轮。

### 5.3 各层职责

**L1 事实抽取**：每次章节保存触发，用便宜模型抽取出 `state_changes / events / new_entities / relation_changes / foreshadow_triggers`，结构化入库。

**L2 规则引擎**（确定性、零成本）：

| 规则类 | 示例 |
|--------|------|
| 状态单调性 | 修为不可倒退、已死角色不可行动、被销毁道具不可再现 |
| 数值合理性 | 年龄 = 出生 + 故事时间；身高/体重不突变 |

| 唯一性 | 同一时刻一个角色只能在一个地点；唯一道具不可被多人持有 |
| 持有关系 | 道具持有者必须健在；持有变更须有显式事件 |
| 称呼一致 | A 对 B 的称呼一旦确立，不应无理由突变 |
| 地理可达 | 跨地点移动必须有合理时间间隔（基于世界观速度） |
| 生死状态 | 死亡角色不可"复活"，除非世界观规则允许 |
| 时序逻辑 | 事件 B 依赖事件 A，B 不能早于 A |

每条规则是独立函数，可由用户启用/禁用/自定义阈值。

**L3 图谱推理**：把 Story Bible 维护成**时序知识图谱**（节点=实体，边=关系，边带时间区间），做：

- 因果链断裂检测：B 依赖 A 的结果，但 A 未发生 / 已被推翻
- 关系冲突：A 同时是 B 的师父和敌人，无关系转换交代
- 位置冲突：同一时间点同一角色出现在两个互斥地点
- 能力体系一致性：突破等级所需条件未满足
- 伏笔回收路径可达性：回收章节前所有 setup 是否齐备

技术：MVP 阶段用 PostgreSQL + 邻接表即可，规模化后再迁 Neo4j。

**L4 LLM 语义层**：只处理软矛盾——

- 性格突变（冷静角色突然暴躁，无铺垫）
- 动机不合理（行为违背欲望/弱点设定）
- 文风跳脱（古风作品出现现代词）
- 情感曲线断裂
- 对话不像该角色（口癖丢失、用词跑偏）

调用前 L1-L3 已过滤硬伤。Prompt 强制 JSON + 引用原文证据 + 置信度评分。

**L5 反馈学习**：每条问题三种用户操作——采纳修复 / 标记为有意 / 忽略此规则。反馈用于：

1. 调整规则置信度阈值（误报多的规则自动降权）
2. 训练专属"白名单分类器"
3. 沉淀为 Prompt 自动迭代的训练样本

### 5.4 触发时机

| 时机 | 跑哪些层 | 体验 |
|------|---------|------|
| 写作中（实时） | L2 轻量规则 | 编辑器内联红线 |
| 章节保存后 | L1 + L2 + L3 | 后台异步，1-2 秒 |
| 用户主动检查 | L1 - L4 全跑 | 进度条，数十秒 |
| 每 5 章 | L3 + L4 全局 | 后台跑 |

### 5.5 输出与定位

每条问题必须给出：

```json
{
  "id": "ISSUE-001",
  "severity": "high|medium|low",
  "layer": "L2",
  "type": "等级倒退",
  "entities": ["林夜"],
  "location": {"chapter_id": 47, "paragraph": 23, "char_offset": [120, 180]},
  "evidence_quote": "林夜运转练气期心法...",
  "conflict_with": {"chapter_id": 35, "fact": "林夜突破筑基期"},
  "suggestion": "改为'筑基期心法'，或前置补充被废修为事件",
  "confidence": 0.95,
  "auto_fixable": true
}
```

`auto_fixable=true` 支持一键应用，其他需用户审阅。

### 5.6 评估指标（一致性引擎自身被评估）

- 召回率 ≥ 85%（金标问题集）
- 精确率 ≥ 70%（避免被用户关掉）
- 定位准确率 ≥ 90%（精确到段落）
- 单章检查 Token 成本 < 5000
- P95 延迟 < 10s

## 六、长程记忆系统

### 6.1 分层摘要

```
L0  原始正文     全文存储
L1  段落摘要     每 ~500 字一条，向量化（bge-m3）
L2  章节摘要     结构化（事件 / 出场 / 状态变化）
L3  卷摘要       主线推进
L4  全书摘要     <2000 字，动态更新
```

### 6.2 上下文 Token 预算（写新章节时）

```
[作品元信息]                       200
[L4 全书摘要]                      800
[L3 当前卷]                        500
[L2 最近 3 章]                    1500
[Bible 相关实体（图谱召回）]      2000
[L1 向量召回相关段落]             2000
[当前章纲 + 写作要求]             1000
─────────────────────────────────
                            ~8000 tokens
```

低于预算优先填 L2，富裕时补 L1。

### 6.3 摘要更新流水线（章节保存后异步触发）

1. 小模型生成 L1 段落摘要 + 向量化入库
2. 中等模型生成 L2 章节结构化摘要
3. **L1 事实抽取** → 更新 Story Bible（触发一致性引擎）
4. 完成一卷后刷新 L3
5. 每 10 章刷新一次 L4

## 七、AI 编排架构（多 Agent）

```
用户请求
   ↓
[路由 Agent]      判断意图（生成/续写/润色/检查/讨论）
   ↓
[上下文 Agent]    检索 Bible + 分层记忆 + 文风（按 token 预算）
   ↓
[生成 Agent]      按任务难度选模型（小/中/大）
   ↓
┌──┴────────┬──────────┬──────────┐
▼           ▼          ▼          ▼
[一致性守卫] [安全守卫] [文风守卫]   并行
└──┬────────┴──────────┴──────────┘
   ↓
[聚合输出]        主结果 + 风险标注 + 备选方案
   ↓
[反馈日志]        用户接受/拒绝 → 数据飞轮
```

### 7.1 模型分级路由（控成本核心）

| 任务 | 模型层级 | 候选 |
|------|---------|------|
| 摘要、命名、分类、事实抽取 | 小 | DeepSeek-V3 / Qwen-Turbo / Haiku |
| 章节正文、续写、润色 | 中 | DeepSeek-R1 / Sonnet / Qwen-Max |
| 大纲设计、深度一致性分析 | 大 | Claude Opus / GPT-4.1 / Gemini Pro |
| Embedding | 专用 | bge-m3 / text-embedding-3 |

经验值：分级路由 + Prompt Caching + 语义缓存 + 摘要压缩，单字成本可降到不分级的 **1/3 ~ 1/5**。

## 十一、核心业务流程

### 11.1 从零创建小说

```
创建作品 → 输入题材 → AI 生成简介 → AI 生成世界观
       → AI 生成主要角色 → AI 生成总纲
       → AI 生成卷纲 → AI 生成章纲
       → 用户确认 → 进入章节写作
```

### 11.2 章节写作（带一致性闭环）

```
选择章节 → 加载章纲 + Bible 相关实体 + 前文摘要
       → RAG 检索相关记忆 → 上下文组装（按 token 预算）
       → AI 流式生成正文 → 用户编辑修改
       → 保存触发：
         ├─ L1 事实抽取 → 更新 Bible
         ├─ L2 章节摘要生成
         ├─ L1 段落摘要 + 向量化
         └─ L2 + L3 一致性检查（异步）
       → 问题列表推送给前端
```

### 11.3 一致性检查全流程

```
用户主动检查 / 章节保存触发
       ↓
L1 事实抽取（如未跑过）
       ↓
L2 规则引擎（毫秒级）
       ↓
L3 图谱推理（毫秒级）
       ↓
L4 LLM 语义检查（数秒）
       ↓
聚合去重 + 置信度排序 → 问题清单
       ↓
用户操作（采纳 / 忽略 / 有意）→ L5 反馈学习
       ↓
更新规则置信度 / 白名单
```

## 十二、Eval 与数据飞轮（原大纲缺失，本版补足）

### 12.1 Eval 体系

**金标测试集**（≥200 case）：

- 给定 Bible + 章纲，生成章节，评估：人物一致性、世界观遵守、字数偏差、文风匹配
- 给定带矛盾的章节，引擎能否定位（Precision/Recall）
- 给定原文，润色后语义保留度 + 流畅度提升

每次 Prompt 改动、模型升级、版本发布前必跑。

### 12.2 在线指标

- 用户接受率（生成内容被保留比例）
- 重生成率（连续点"再来一次"，越高越烂）
- 章节完成时长（AI 协作前后对比）
- 一致性问题用户确认率（防止过度警告）

### 12.3 数据飞轮

```
用户写作行为
   ↓
[采集] AI 生成 → 用户保留/修改/丢弃 → diff
   ↓
[挖掘] 高频被改句式 → 文风偏好 / 反向 Prompt 优化
       常见拒绝原因   → 守卫 Agent 规则更新
       优质保留段落   → 私有微调样本（用户授权后）
   ↓
[反哺] Prompt 自动迭代 / 模板市场 / 个性化文风模型
```

**这是产品长期壁垒**——竞品可抄功能，抄不走数据。

## 二十、最终产品形态

该系统最终形态是面向长篇小说生产的**智能创作中台**，而不是"AI 写作工具"。

核心能力链条：

```
帮你想故事 → 帮你搭世界 → 帮你做人设
         → 帮你写大纲 → 帮你生成章节 → 帮你润色修改
         → 帮你管理长篇设定 → 帮你检查逻辑矛盾
         → 帮你提升作品质量
```

**三大壁垒**：

1. **Story Bible 数据资产**：用户用得越久，资产越厚，迁移成本越高
2. **一致性引擎**：技术深度，难以被竞品快速复制
3. **数据飞轮**：用户行为反哺 Prompt 和规则，越用越准
